# Sprint TECH-RISK-PROTOTYPE: Continuity Camera入力と反応描画の実証

> **ステータス**: 完了（2026-07-18）

## 目的

iPhone 17 ProをUSB-CでMacBook Airへ接続し、ブラウザがContinuity Cameraの映像を取得して、映像全体の変化へ即座に反応する描画までを一気通貫で検証する。

## ユーザージャーニー

1. 運営者がiPhoneをUSB-CでMacへ接続する。
2. Macでローカルアプリを開き、利用可能なカメラからiPhoneを選ぶ。
3. カメラ担当が参加者へiPhoneを向ける。
4. 参加者が動くと、ライブ映像上の発光・粒子量と動きメーターが変化する。
5. 運営者が、入力、解析、描画の成立または失敗理由を画面から判断できる。

## アーキテクチャ

```text
iPhone 17 Pro
  -> USB-C / Continuity Camera
  -> Chrome MediaDevices API
  -> Video element
  -> 低解像度Canvasのフレーム差分解析
  -> 表示Canvasの反応描画
```

特徴解析はブラウザAPIから独立した純粋関数とし、合成ピクセルデータによる単体テストを先に作る。カメラ映像は保存・送信しない。

## 成功条件

- カメラ権限の状態と失敗理由が画面に表示される。
- 利用可能なカメラを再取得し、選択して開始できる。
- 入力映像がブラウザ内へ表示される。
- 動きの増減が数値メーターとCanvas演出へ反映される。
- 映像停止時に演出が減衰する。
- 特徴解析の自動テストが合格する。
- 実機確認結果が本書に記録される。

## 残るリスク

### 実機検証

USB接続したiPhone 17 ProをChromeのカメラ候補から選択し、次を確認した。

1. iPhone映像がライブ表示された。
2. カメラ前の動きに応じて「動き」メーターが変化した。
3. 動きに応じた発光と粒子が表示された。
4. 動作中も描画が60 FPSを維持した。
5. Mac画面で全画面表示へ切り替えた後も映像とエフェクトが動作した。

[FACT: 実機受け入れ 2026-07-18 → iPhone 17 Pro、Chrome、M4 MacBook Airで映像入力、動き連動エフェクト、動作中60 FPS、全画面表示をユーザー確認]

### 性能問題と解決

最初の実機確認では描画が4 FPSまで低下した。負荷要因を一つずつ除外した結果、全画面ぼかし除去後は5 FPS、全画面色調フィルター除去後は7〜30 FPS、粒子ごとの`shadowBlur`除去後は動作中も60 FPSとなった。主因は、粒子数に比例して個別のぼかし処理が増える描画方式だった。[FACT: issues/2026-07-18-canvas-particle-shadow-performance.md → 調査経緯、原因、対応、実機検証]

粒子と加算合成による発光は維持し、個別のぼかし影だけを廃止した。[FACT: public/app.js:303-356 → 粒子生成と加算合成描画]

## 結論

ブラウザがUSB接続のContinuity Cameraを映像入力として利用でき、低解像度特徴解析から半抽象演出までをローカルで接続できた。完成演目へ進むための入力・解析・描画経路は実機で成立した。
